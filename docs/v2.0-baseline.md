# NovelAgent 2.0 Baseline and Stabilization Charter

## Release meaning

Version 2.0 freezes the current NovelAgent feature set as the starting point for focused stabilization work. It preserves the accumulated StoryProject, semantic-state, Memory V2, persistence, delivery, validation, repair, recovery, context-budget, and provider-evidence capabilities.

The `v2.0` tag does not claim that unattended long-form generation is production-ready. It marks a reproducible stage boundary: subsequent 2.x work should optimize, simplify, verify, or repair existing behavior instead of expanding the product surface.

## Evidence behind the baseline

The repository retains the earlier novel archive and the ten-chapter comparison corpus under `exports/`. A real rewrite exercise proved that the current system can improve local prose quality, atmosphere, physical action, tactical detail, and the presentation of ability costs.

The same comparison also exposed delivery-level regressions:

- overlapping or repeated narrative segments can reach formal prose;
- character identity and motivation can drift between chapters;
- numeric and roster state can roll back or split across competing versions;
- a run can complete and commit without proving that the multi-chapter reading experience is coherent;
- provider uncertainty, timeout, budget, and recovery behavior still interrupt operator workflows;
- context growth and compression remain risks for dozens or hundreds of chapters.

The exact pipeline stage that first introduced each duplicated narrative segment is not yet proven. The v2.0 evidence establishes the symptoms and their effect on the deliverable, not an unverified implementation-level root cause.

## Release validation

The v2.0 release candidate passed the local deterministic gates:

- `python -m pytest -p no:cacheprovider tests`: 1,377 passed, 7 skipped;
- `python -B scripts/smoke_v1.py --skip-tests`: passed, including 21/21 preflight checks.

These results verify the retained code contracts and local dry-run path. They do not replace the real-provider and multi-chapter content-quality evidence described above.

## Stabilization priorities

The following work is in scope for the next 2.x releases, in priority order:

1. Final-artifact integrity: reject overlapping drafts, repeated scenes, duplicate hooks, and accidental append-versus-replace repairs.
2. Cross-chapter state authority: make character identity, relationships, numeric systems, rosters, inventory, locations, time, and resolved setup/payoff state auditable and consistent.
3. Trustworthy quality gates: distinguish workflow completion from validation success and delivery readiness.
4. Safe recovery: make provider uncertainty and failed chapters resumable or discardable without corrupting committed prose or forcing the operator to inspect raw artifacts.
5. Long-horizon context: bound context growth without silently deleting facts that later chapters require.
6. Operational viability: expose per-chapter calls, tokens, elapsed time, repair count, failure waste, and human interventions.
7. Maintainability: reduce interacting exceptional paths and express the chapter lifecycle as a testable state machine.

## Stabilization exit criteria

A later 2.x release should not claim the stabilization cycle complete until retained evidence demonstrates all of the following:

- deterministic regression fixtures catch the real duplicate-scene, character-identity, and numeric-state failures found in the comparison corpus;
- the final prose bytes are bound to the exact artifact that passed the delivery-level gate;
- a fixed multi-chapter corpus completes without scene duplication, state rollback, or unsafe recovery;
- interrupted calls can be diagnosed and recovered through a user-facing command without manual artifact surgery;
- long-horizon context tests preserve declared hard facts while remaining within configured budgets;
- success reports include cost, latency, call-count, repair, and intervention evidence.
